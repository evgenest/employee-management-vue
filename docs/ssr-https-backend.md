# SSR и HTTPS-бэкенд: проблема и решения

## Суть проблемы

При использовании Nuxt с SSR (серверным рендерингом) `useAsyncData` выполняется **дважды**:

- **На сервере** (Node.js) — при первой загрузке страницы или F5
- **На клиенте** (браузер) — при навигации через `<NuxtLink>`

Если бэкенд работает на `https://localhost` с dev-сертификатом (например, ASP.NET Core dev-cert), Node.js **не доверяет** ему по умолчанию и запрос падает. Браузер при этом работает нормально, потому что пользователь уже принял исключение для localhost.

---

## Решения

### 1. `server: false` в `useAsyncData` ✅ реализовано

Данные всегда загружаются на клиенте, SSR-запрос к бэкенду не выполняется.

```ts
// pages/index.vue
const { data, refresh } = await useAsyncData('employees', () => api.getAll(), { server: false })
```

**Плюсы:** просто, предсказуемо, не требует настройки сертификатов.  
**Минусы:** нет данных в первом HTML-ответе (таблица рендерится после загрузки JS).  
**Когда подходит:** внутренние инструменты без SEO-требований.

---

### 2. `NODE_EXTRA_CA_CERTS` — добавить сертификат бэкенда

Node.js будет доверять указанному CA-сертификату наравне со стандартными.

В `.env`:
```env
NODE_EXTRA_CA_CERTS=./certs/backend-ca.pem
```

Экспорт сертификата из ASP.NET Core:
```bash
dotnet dev-certs https --export-path ./certs/backend-ca.pem --format Pem --no-password
```

**Плюсы:** правильный подход, SSR работает полноценно.  
**Минусы:** нужно управлять сертификатом и добавить `certs/` в `.gitignore`.  
**Когда подходит:** если SSR важен и есть доступ к сертификату бэкенда.

---

### 3. `NODE_TLS_REJECT_UNAUTHORIZED=0` — отключить проверку

Полностью отключает SSL-валидацию в Node.js.

В `.env`:
```env
NODE_TLS_REJECT_UNAUTHORIZED=0
```

**Плюсы:** одна строка, мгновенно работает.  
**Минусы:** небезопасно — Node.js принимает любой сертификат, включая поддельные.  
**Когда подходит:** только локальная разработка, никогда не в production.

---

### 4. Кастомный агент через Nuxt-плагин

Позволяет гибко настроить поведение `$fetch` только на сервере.

```ts
// plugins/fetch.server.ts
import { Agent } from 'node:https'
import { readFileSync } from 'node:fs'

export default defineNuxtPlugin(() => {
  const ca = readFileSync('./certs/backend-ca.pem')

  globalThis.$fetch = $fetch.create({
    fetchOptions: {
      dispatcher: new Agent({ ca })
    }
  })
})
```

Или с отключённой проверкой (dev-only):
```ts
dispatcher: new Agent({ rejectUnauthorized: false })
```

**Плюсы:** изолировано в одном файле, не затрагивает глобальный Node.js.  
**Минусы:** сложнее, чем env-переменная.  
**Когда подходит:** если нужна тонкая настройка или несколько разных агентов.

## В production

В production используется CA-подписанный сертификат, которому Node.js доверяет по умолчанию. Там `server: false` можно убрать, и SSR будет работать без дополнительной настройки.
